# 远端文件列表与缩略图重构说明

本次修改彻底移除了原先通过 `ThumbnailsLoadingService` 启动本地 HTTP 服务、再由 Glide 请求远端文件的缩略图链路，改为应用内部统一的异步缩略图仓库。

## 文件列表显示

- **列表模式**：保留原有文件名、修改时间、大小、选择、移动、排序和文件操作行为；图片与视频文件左侧显示实际缩略图，其他文件仍显示文件图标。
- **图标模式**：缩略图在上、文件名在下；列数根据 RecyclerView 可用宽度自动计算。
- 图标模式提供 **超大、大、中、小** 四档，目标列宽分别为 300dp、220dp、156dp、112dp。
- 图片按照自身宽高比调整预览高度；同一行由 GridLayoutManager 按最高项目确定行高。
- 显示模式按完整远端路径绑定。未保存的路径统一使用列表模式；只有图标模式会写入 SharedPreferences，切回列表时会删除该路径的记录。

## 新缩略图链路

核心类：`app/src/main/java/ca/pkay/rcloneexplorer/thumbnails/ThumbnailRepository.java`

- 图片使用 `BitmapFactory` 按采样率解码，并处理 EXIF 旋转/镜像方向（Android 7.0 及以上）。图片会完整读取到临时文件，但只允许读取不超过设置中“图片源文件完整下载上限”的源文件；即使远端未提供可靠大小，复制过程中也会再次执行该上限。
- 视频首先使用 Android `MediaMetadataRetriever` 在总时长约三分之一处提取；只要平台提取返回空或抛出异常，就统一回退到应用内嵌的精简版 FFmpeg，不按扩展名特判，因此 WMV 以及平台不支持或解析失败的 AVI、MOV、MKV、MPEG-TS 等格式都会尝试兜底。
- FFmpeg 使用自定义 `AVIOContext`。其 `read_packet`/`seek` 回调经 JNI 转换为 Java `readAt(position, ...)`，再进入现有 `rclone cat --offset ... --count ...` 范围读取和 4–8 MiB 读前缓存；支持 `SEEK_SET/CUR/END` 与 `AVSEEK_SIZE`。整个过程不创建完整远端视频临时文件。SAF 兜底使用可定位的文件描述符和 `FileChannel`。
- 精简 FFmpeg 仅编译 LGPL 的 `libavformat`、`libavcodec`、`libavutil`、`libswscale` 和常用视频容器/软件解码器，关闭程序、网络、编码器、滤镜、设备、GPL/非自由组件及外部压缩库。详细设计见 `docs/FFMPEG_THUMBNAIL_FALLBACK.md`。
- 生成缩略图最长边为 **1280 px**；无透明通道时以 JPEG 质量 92 保存，有透明通道时保存为 PNG。
- MIME 不可靠时，会使用常见图片、视频扩展名进行兜底识别。
- 同一文件的并发请求会合并；RecyclerView 项复用后会取消未开始的请求，防止快速滚动产生无意义的远端读取。

## 本地持久化缓存

- 缓存根目录仍为应用私有 `thumbnail_cache`，但图片与视频分别存放在 `thumbnail_cache/images` 和 `thumbnail_cache/videos`；工作目录同样拆分为 `thumbnail_work/images` 与 `thumbnail_work/videos`，不会暴露给其他应用。
- 根目录结构版本为 `split-media-thumbnail-v1`；图片缓存版本为 `image-adapt-extract-v1`，视频缓存版本为 `video-third-dynamic-v1`。本版本不兼容、不迁移旧缓存：根结构或对应媒体类型版本不匹配时直接清空相应目录并重新建立版本标记。
- 图片与视频分别维护磁盘锁、缓存代际、内存 LRU、清理调度、统计快照和持久化状态。清空、导入或版本失效只递增对应媒体类型的代际，不会使另一类正在进行的请求失效。
- 缓存键包含对应媒体类型的缓存版本、远端名称、远端类型、文件路径、大小和修改时间；文件发生变化后会自然命中新键。图片与视频即使源信息相同，也因版本和目录不同而无法互相命中。
- 图片和视频各自拥有磁盘缓存上限与清理目标，默认均为 2048 MB 和 1536 MB。任一类型超过自身上限时，只按现有 LRU 时间戳逻辑清理该类型到自己的目标值，另一类型不参与统计或删除。
- 图片和视频分别拥有独立的全局缩略图开关；关闭某一类型只停止并失效该类型的显示与生成任务，不清除其磁盘缓存，也不会影响另一类型。
- 图片和视频的缓存统计分别使用独立的内存快照及 SharedPreferences 状态文件，分别记录 `.thumb` 数量和总占用空间；负缓存标记和临时文件不计入统计。
- 导入/导出归档格式为版本 1，清单中同时记录 `cacheType` 与对应 `cacheVersion`。图片归档仅包含 `image-thumbnails/`，视频归档仅包含 `video-thumbnails/`；类型或版本不一致的归档会在写入前被拒绝。
- 设置页分别提供图片和视频的统计、导入、导出、清空操作。每项操作只锁定和处理自己的缓存目录、内存缓存与持久化状态。
- 损坏缓存会自动删除并重新生成；提取失败会在对应媒体目录写入 12 小时负缓存，避免重复读取。
- 图片临时源文件在生成完成后删除；图片和视频工作目录中上次异常退出遗留的临时文件会分别在仓库初始化时清理。

## 远端读取

`Rclone` 提供统一远端文件路径构建方法和带 `--offset` + `--count` 的字节范围流读取方法。Android 平台提取器与 FFmpeg 自定义 AVIO 都复用同一个 `RcloneRangeMediaDataSource.readAt()`，把请求的文件偏移转换成 Rclone 范围读取，因此可以访问容器头部、尾部索引以及目标关键帧附近的数据，而不需要先把视频保存为完整本地文件。

该实现不再需要缩略图 HTTP 服务、随机端口、隐藏鉴权路径或相关前台服务声明。

## 远端媒体串流路径保护

- 文件点击播放所使用的临时 HTTP 串流不再直接暴露在 `http://127.0.0.1:<port>/<文件名>`。每次播放都会在后台使用 `SecureRandom` 生成 32 字节（256 位）随机值，并以 Base64 URL-safe、无填充形式作为单独的 URL 路径段。
- 同一个随机值通过 `StreamingService.SERVE_BASE_URL` 传入 rclone `serve http --baseurl`；播放器和可用性 HEAD 检查只访问 `http://127.0.0.1:<port>/<token>/<文件名>`。rclone 的 base URL 中间件会让不带 token 的根路径及错误前缀返回 404，因此仅知道本地端口不足以读取文件。
- token 只存在于当前串流 Intent、服务进程和交给播放器的临时 URL 中，不写入设置或持久化状态；启动下一次串流会生成全新的 token。应用自己的等待日志只记录端口，不打印完整的受保护 URL。
- 该机制属于本机随机能力 URL，而不是账号体系：获得完整 URL 的播放器在服务存活期间仍可访问对应文件。服务仍只监听 `127.0.0.1`，播放结束后沿用原有逻辑停止服务。

## 文件夹递归预生成

- 当前远端文件夹的工具栏菜单新增“递归生成图片缩略图”和“递归生成视频缩略图”。两项操作分别提交到 `ThumbnailPreGenerationService`，任务按 IntentService 队列串行执行，避免一次性建立无界并发远端读取。
- 服务使用广度优先方式逐层调用现有 `Rclone.getDirectoryContent()`：当前目录内发现的子目录会继续入队，直到遍历完所有层级；使用已访问路径集合避免重复目录。目录列表按批处理，不需要一次把整棵目录树保存在内存中。
- 图片任务只接受 `isImageThumbnailCandidate()`，视频任务只接受 `isVideoThumbnailCandidate()`；每个文件通过 `preGenerateThumbnail(item, CacheType)` 写入对应磁盘缓存。该入口不解码显示 Bitmap、不写入另一类型的内存缓存，也不会更新已存在缓存文件的 LRU 时间戳。
- 预生成仍遵守对应媒体类型的全局开关、缓存代际、负缓存、图片源文件下载上限、视频随机访问提取以及各自的磁盘容量清理。运行中关闭本类型开关会停止该任务；另一类型开关和缓存不受影响。
- 服务以前台通知显示已扫描文件夹数和已处理的匹配文件数，并持有 CPU/Wi-Fi 锁；完成后报告新生成、已有缓存、失败文件和读取失败目录数量。完成广播只刷新当前界面中相同媒体类型的条目。

## 设置变化

- 删除原有的图片与视频共用“显示缩略图”总开关，改为图片和视频两个相互独立、默认开启的全局缩略图开关。关闭图片开关只停止图片缩略图的显示与生成，关闭视频开关只停止视频缩略图的显示与生成。
- “图片缩略图”和“视频缩略图”设置分组分别提供缓存上限、清理目标、当前数量与空间、导入、导出和清空功能。
- “图片源文件完整下载上限”保留在图片缩略图分组中，仅约束图片；视频不受该选项影响，因为视频不执行完整源文件下载。
- 所有容量设置使用 MB 整数输入方式；图片与视频的默认磁盘缓存上限均为 **2048 MB**，清理目标均为 **1536 MB**。
- 缓存版本变化时不执行旧缓存迁移；根结构不兼容时清空旧结构，单一媒体缓存版本不兼容时仅清空对应媒体目录。

## 建议验证

- 对 `moov` 位于文件尾部的 MP4、MKV/WebM 和较大视频验证三分之一处取帧，确认能看到针对头部、尾部及目标样本区域的范围读取，而没有完整视频临时文件。
- 对大小已知且超过图片上限的图片验证直接拒绝；对大小未知的图片验证复制超过上限时会中止并删除临时文件。
- 在设置页分别修改图片与视频的缓存上限和清理目标，确认均以 MB 显示、分别持久化；图片源文件完整下载上限只出现在图片分组。
- 分别制造图片缓存超限和视频缓存超限，确认清理后只有对应目录下降到自己的目标值，另一目录的文件、统计与内存命中不受影响。
- 分别导出图片与视频缓存，再交叉导入，确认类型不匹配时在写入前拒绝；同类型导入只新增或覆盖对应目录。
- 分别执行图片清空和视频清空，确认另一媒体类型的磁盘文件、统计、负缓存和正在进行的生成请求不受影响。
- 对直接视频播放抓取本地请求，确认播放器 URL 含 43 字符左右的随机 token 路径段，裸根路径和错误 token 返回 404，正确 token 支持 HEAD、GET 与 Range 请求。
- 在含多级子目录、混合图片和视频的远端分别执行两种递归任务，确认只生成目标类型、已存在缓存不被重写为最新 LRU、关闭其中一个全局开关只停止对应任务。
- 在具备 Android SDK 和依赖缓存/网络的开发机上执行最终 Gradle 构建及真机验证。
